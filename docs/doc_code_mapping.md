# documentCode 突合表 — 仕様書 / Lievito 認識 / 実 API 観測

## 目的

新規 FB で未知の書類コードに遭遇したとき、 同じ書類を別の派生コードで取りこぼし続ける
「モグラたたき」を防ぐための事実記録。

- **仕様書**: 「特許情報標準データ 仕様書 v1.13」(JPO/INPIT 公式) 4-01 出願マスタ・4-18 共有データベース（審判）の「別紙 中間コード一覧」
- **Lievito 認識**: `generator/dates.py` の TYPE_A_DOCS / TYPE_B_DOCS / TYPE_A_KEYWORDS で扱っているもの
- **実 API 観測**: `inventory/doc_history_collected/*.json` (84 件) の `bibliographyInformation[].documentList[].documentCode` 集計

## Lievito の識別経路 (重要)

書類タイプによって識別ロジックが異なる:

| 経路 | データソース | 識別方法 | 対象書類 |
|---|---|---|---|
| ① Type A 主経路 | `doc_xmls/{appno}/_summary.json` | **日本語の書類名** (`TYPE_A_KEYWORDS`) | 拒絶理由通知書 / 拒絶査定 / 補正の却下の決定 / 特許査定 / 前置報告書 / 当審拒絶理由通知書 |
| ② Type A 補助 | `inventory/zenchi_drafting/{appno}.json` | 補助ソース取得 | 前置報告書 (A913) のみ。 理由: A913 の起案日 (= 報告書本文「作成日」) が API では取れないため (API legalDate は発送日) |
| ③ Type B | `doc_history.json` | **`documentCode`** (`TYPE_B_DOCS`) | 意見書 / 手続補正書 / 誤訳訂正書 / 翻訳文 / 応対記録 / 審判請求書 / 上申書 など |

**重要**: Type A は **書類名 (日本語)** で識別しているため、 documentCode が仕様書定義
(A191/A01 など) と Lievito 認識 (A502/A03) で乖離していても、 同じ日本語名で来る限り
拾える。

将来観測されたとき、 想定どおりに拾えるかは要検証:

- A132 / A133 (拒絶理由通知書 派生)
- A192 (補正の却下の決定 派生)

- A03 (特許査定 旧来) — 同上

## 更新手順

新規 FB で未知の書類コードが判明したら:

1. 本ファイルにエントリを追加 (仕様書名 / Lievito 認識 / 実 API 観測 / 経緯対象)
2. `generator/dates.py` の TYPE_A_DOCS / TYPE_B_DOCS マップにコードを追加
3. Type B なら `_from_doc_history_json` の dispatch にも追加
4. 動作確認 → commit → push

仕様書原典は VPS には置かない (ライセンス・容量の観点)。 仕様書本体が必要な場合は
ユーザーが claude_user_io 経由で参照。
